## TL;DR (3 lines)

- **What it is:** Open-source clone of DeepWiki (the closed-source AI-wiki product) — turns any **GitHub / GitLab / BitBucket** repository into an interactive wiki with Mermaid diagrams + a RAG-powered "Ask" feature + multi-turn "DeepResearch". MIT, by Sheing Ng. Stack: **Next.js 15 + React 19 + Mermaid** frontend + **FastAPI + adalflow + FAISS** backend.
- **How its KB works:** Repos are cloned via `download_repo` (HTTPS/PAT-protected), traversed by `read_all_documents`, embedded via configurable embedder (OpenAI / Google AI / Ollama), indexed in **FAISS** via adalflow's `FAISSRetriever`, and served by `DatabaseManager`. Wiki pages are LLM-generated with a hierarchy schema (`WikiSection` → `WikiPage`), then cached. Mermaid diagrams are auto-generated from architecture analysis.
- **Verdict:** Pick when you want a **server-shaped, multi-language wiki-compiler** that runs on your infra and works against any Git host. Skip if you want a Claude-Code-plugin-shape (use Understand-Anything) or want pure-codebase-graph reasoning without a wiki/chat layer.

## KB Architecture

### Storage
- **Vector store:** **FAISS** (`faiss-cpu>=1.7.4`) via [`adalflow.components.retriever.faiss_retriever.FAISSRetriever`](https://github.com/SylphAI-Inc/AdalFlow). Cohort second-largest FAISS user (mem0, LightRAG, AstrBot, **deepwiki-open**).
- **Graph store:** *None.* DeepWiki-open is wiki-shaped; relationships live in the rendered Mermaid diagrams, not in a graph DB.
- **Metadata / structured:** Local filesystem under `adalflow_default_root` (configured per-deployment); cached wiki pages persisted as JSON (`WikiCacheData`).
- **Object / blob:** Local filesystem; cloned repos live in a working directory.

### Ingestion / Extraction
- **Source types accepted:** **GitHub / GitLab / BitBucket** repos via `download_repo` with optional PAT for private repos. Per-host file fetchers (`get_github_file_content`, `get_gitlab_file_content`, `get_bitbucket_file_content`).
- **Chunking strategy:** `transform_documents_and_save_to_db` runs adalflow's pipeline; token budget capped at `MAX_INPUT_TOKENS = 7500` to stay below the 8192-token embedding model limit.
- **Entity / fact extraction:** **LLM-driven** — `WikiStructureModel` schema is generated by an LLM call (system prompts in `prompts.py`); pages are then filled in iteratively. Mermaid diagrams generated as part of architecture analysis.
- **Schema:** Pydantic models — `WikiPage`, `WikiSection`, `WikiStructureModel`, `WikiCacheData`, `WikiExportRequest`, `RepoInfo`. Cohort-second-most-typed wiki schema after Understand-Anything's 35-edge Zod schema.

### Retrieval
- **Modes:** **Vector-only** via FAISS; no BM25 / hybrid. The "Ask" feature uses RAG on the indexed repo, "DeepResearch" runs **multi-turn iterative retrieval** to investigate complex topics — multi-turn research loop is a notable cohort entrant.
- **Reranker:** None.
- **Top-k defaults:** Configurable per request via adalflow.
- **Context assembly:** `RAG_SYSTEM_PROMPT` + `RAG_TEMPLATE` from `prompts.py`; conversation history maintained via custom `CustomConversation` / `DialogTurn` / `UserQuery` / `AssistantResponse` dataclasses (workaround for an adalflow `list assignment index out of range` bug, per code comments).

### Memory model
- **Tiers:** None — single wiki cache per repo. No conversational memory across users.
- **Bi-temporal:** No.
- **Self-update mechanism:** Wiki regenerated on demand; no incremental update tracking. (Compare: Understand-Anything's `PostToolUse` git-commit hook fires auto-update.)
- **Decay / forgetting:** Cache invalidation is manual.

### MCP / connectors
- **MCP server exposed:** **No** native MCP. (`regenrek/deepwiki-mcp` is a separate ★1.3k repo that wraps the DeepWiki experience as MCP.)
- **MCP client used:** **No.**
- **Native connectors:** **3 Git hosts** (GitHub, GitLab, BitBucket) with PAT support. **5+ LLM providers**: Google Gemini, OpenAI, OpenRouter, Ollama (local), **Azure OpenAI, AWS Bedrock, DashScope** (per `azureai_client.py`, `bedrock_client.py`, `dashscope_client.py`, `google_embedder_client.py`, `ollama_patch.py`, `openai_client.py`, `openrouter_client.py`).
- **Tool-call surface:** REST API (`/lang/config`, `/auth/status`, `/auth/validate`, `/models/config`, `/export/wiki`, `/local_repo/structure`, …) consumed by the Next.js frontend.

### Notable design choices
- **DeepWiki clone** — explicitly named as "my own implementation attempt of DeepWiki" by Sheing Ng. The closed-source DeepWiki (deepwiki.com) is the reference; this is the OSS interpretation.
- **adalflow** as the RAG framework — relatively new alternative to LangChain (developed by SylphAI Inc.). Cohort first to use adalflow.
- **3 Git hosts + PAT auth** — broader-than-Understand-Anything coverage (UA is local-only). Private-repo support is first-class.
- **Mermaid as the diagram primitive** — `mermaid==^11.4.1` in package.json + `svg-pan-zoom` for interactive panning. Matches the way DeepWiki itself shows architecture diagrams.
- **DeepResearch multi-turn loop** — 3-stage iteration with distinct prompts: `DEEP_RESEARCH_FIRST_ITERATION_PROMPT` → `DEEP_RESEARCH_INTERMEDIATE_ITERATION_PROMPT` → `DEEP_RESEARCH_FINAL_ITERATION_PROMPT` ([`api/simple_chat.py:23-25`](https://github.com/AsyncFuncAI/deepwiki-open/blob/main/api/simple_chat.py)). Separate from the single-turn Ask flow. Cohort first to ship multi-turn DeepResearch under that name in a wiki context (alongside onyx's Deep Research, which is more general-purpose).
- **10 README languages** — EN / zh / zh-tw / ja / es / kr / vi / pt-br / fr / ru. Most languages in cohort.
- **Custom Conversation classes** — code comments explicitly note a fix for an adalflow bug (`list assignment index out of range`) by reimplementing `Conversation` / `DialogTurn` locally. Cohort first to ship a documented workaround for an upstream framework bug.
- **`DEEPWIKI_EMBEDDER_TYPE` env var** — runtime switch between OpenAI / Google AI / Ollama embeddings. Useful when the LLM provider doesn't match the embedding provider (e.g., Gemini chat + OpenAI embeddings).
- **Single-author project** — Sheing Ng built and maintains it; ★16k from a one-person OSS effort.
- **MIT licensed** — pure permissive.

## Tradeoffs

**Pros:**
- **3-Git-host + PAT support** — broadest Git-source coverage in the wiki-compiler category.
- **Mermaid diagrams as a first-class output** — visual architecture without bolting on a separate diagramming step.
- **DeepResearch multi-turn flow** alongside single-turn Ask — depth + breadth in one product.
- **Server-shaped + Docker** — runs on your infra, not as a host-tool plugin.
- **5+ LLM providers + 3 embedder types** with runtime switch — vendor-agnostic.
- **10 README languages + next-intl** — broad geographic reach.
- **Pydantic schemas** for the wiki structure — typed export / cache.
- **Single-binary-ish Docker deployment** with `docker-compose.yml`.

**Cons:**
- **Single-author project** — bus-factor 1; sustained development depends on Sheing Ng.
- **Single-author `CustomConversation` workaround** suggests adalflow has rough edges; coupling to a less-mature framework is a risk.
- **No incremental update** — wiki must be regenerated, no git-commit-hook-shape staleness check (compare: Understand-Anything's auto-update hooks).
- **No graph backend** — relationships live in Mermaid diagrams (rendered, not queryable).
- **Vector-only retrieval** — no hybrid BM25, no reranker.
- **No MCP** — third-party `regenrek/deepwiki-mcp` exists but is separate.
- **No conversational memory across users** — each wiki + Ask session is independent.
- **`MAX_INPUT_TOKENS = 7500` hard limit** — embedding-model-specific assumption baked in.

## When to use it

- **Good fit:** teams wanting a self-hosted DeepWiki-clone for internal repos; products that need to wikify GitHub / GitLab / BitBucket sources at the *server* tier (not as a coding-agent plugin); deployments wanting Mermaid-diagram-first architecture views with multi-language UI.
- **Bad fit:** Claude-Code-plugin-shape consumers (use [`Lum1104/Understand-Anything`](surveys/Lum1104__Understand-Anything.md) instead); projects needing graph-DB queries over the codebase; products needing MCP server out of the box; heavy-multi-tenant hosted SaaS (no auth-as-platform, single-deployment).
- **Closest alternative:** [`Lum1104/Understand-Anything`](surveys/Lum1104__Understand-Anything.md) — same wiki-compiler category but plugin-shaped instead of server-shaped, and Zod-typed 35-edge KG vs DeepWiki's Pydantic Wiki schema. [`Tencent/WeKnora`](surveys/Tencent__WeKnora.md)'s Auto-Wiki feature is the most direct conceptual peer (also turns docs/repos into a wiki) but as one feature inside a kb-app platform; DeepWiki-open is the wiki-as-the-whole-product variant. [`AgriciDaniel/claude-obsidian`](https://github.com/AgriciDaniel/claude-obsidian) (queued in candidates) is the Obsidian-vault variant of the Karpathy LLM wiki pattern.
